an article about branching strategies

• Note of reflection
if a team doing continuous delivery of software it's better they use simpler workflow like github.
but if they building software that is versioned, or if they need to support multiple versions of software, then it's better they use git-flow
• why git?
in the past, merging/branching have been difficult tasks and some thing you can only do every once in a while. but now because of git and github they are simple and routine tasks.
• Decentralized but centralized
people who work on the same project, in addition to their direct relationship that they have with origin , they can form a team together and work on a big feature .
• the main branches
The central repo holds two main branches with an infinite lifetime:
master
develop
The master branch should be familiar to every Git user.
Parallel to the main branch, there is also an another branch called develop
we consider the master branch as the main branch
We consider the develop branch as the branch where the latest development changes for the next version are submitted.
Some would call this the “integration branch”.
When the source code in the develop branch reaches a stable point and is ready to be released,
all of the changes should be merged back into master
each time when changes are merged back into master,
this is a new release
• supporting branch
next to main branches and develop branch, our development model uses a variety of other branches too.
unlike the main branches, these branches always have a limited life time. and they will removed eventually.
the diffrent types of branches we may use are:
feature branches
release branches
hotfix branches
• feature branches
branch off from develop and Must merge back into develop.
branch cannot have the following names: master, develop, release-*, or hotfix-*
Feature branches are used to develop new features for the upcoming or a distant future release.
feature branch exists as long as the feature is in development,
But in the end, it will be merged into the develop branch.or discarded
• Creating a feature branch
When we want to starting work on a new feature, we should branch off from the develop branch.
• Incorporating a finished feature on develop
Finished features may be merged into the develop branch to add them to the upcoming release.
The --no-ff flag causes the merge to always create a new commit object.
This avoids losing information about the historical existence of a feature branch and groups together all commits that together added the feature.
Using the --no-ff flag, we can see from the Git history which of the commit objects together have implemented a feature
• Release branches
each branch should merge back to develop or master branch.
We should name branches like this:
release-*
Release branches support preparation of a new production release. they allow for minor bug fixes and preparing meta-data for a release.
This makes the develop branch cleared to receive features for the next release.
when deveolp reflects the desired state of the new release. we make new release branch from develop branch.
It is at the start of a release branch that the upcoming release gets assigned a version number
• Creating a release branch
Release branches are created from the develop branch.
After creating a new branch and switching to it, we bump the version number.
bump-version.
sh is a fictional shell script that changes some files in the working copy to reflect the new version.
This new branch may exist there for a while, until the release may be rolled out definitely.
bug fixes may be applied in this branch . Adding large new features here is strictly prohibited.
• Finishing release branch
When the state of the release branch is ready to become a real release, Some actions must be taken.
First:
the release branch is merged into master
Next,
that commit on master must be tagged for easy future reference to this historical version.
Finally,
the changes made on the release branch need to be merged back into develop, so that future releases also contain these bug fixes.
• Hotfix branches
each branch that branching off from main branch , must merge back into develop or main branch.
Branch naming convention:
hotfix-*
Hotfix branches are like release branches. they both meant to prepare for a new production release.
we use hotfix branch when critical bug in a production version must be resolved immediately.
• Creating the hotfix branch
we branch off hotfix branches from master branch
for example :
when changes on our production develop are yet unstable.
we branch off a hotfix branch and start fixing the problem.
• Finishing a hotfix branch
when bugs fixed we should merge back hotfix into master.
we also need to merge it back into develop branch. This is completely similar to how release branches are finished.
when a release branch currently exists, the hotfix changes need to be merged into that release branch, instead of develop.
when we merge back bugfix into relase brach, its look like bugfix being merged into develop too.
• Summary
Although there is nothing very shocking about branching model, but its very useful to over projects.
and its allows team members to develop a shared understanding of the branching and releasing processes.
